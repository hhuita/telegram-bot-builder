# Attribution Model — Модель атрибуции источников

## Идея

Каждый пользователь приходит откуда-то. Нужно знать не только откуда,
но и насколько этому источнику можно доверять.

Два типа данных:
- **Deep link** — объективно, из URL. Достоверность 100%.
- **Survey** — со слов пользователя, из опроса при онбординге. Достоверность ~70%.

---

## Проблема текущего подхода

`deep_link_param = "direct"` означает что пользователь пришёл без ссылки.
Но это не значит что источника нет — он просто неизвестен автоматически.

Пример: человек увидел рекламу в Instagram, потом нашёл бота через поиск
Telegram и написал `/start` без параметра.
- `deep_link_param = "direct"` — технически верно
- Реальный источник = Instagram — только опрос это покажет

---

## Решение — два поля атрибуции

```
bot_users
  deep_link_param     ← автоматически из /start параметра
  referrer_id         ← автоматически из ref_-префикса
  acquisition_source  ← из опроса (зарезервированная переменная)
  acquisition_method  ← "deeplink" | "survey" | "deeplink+survey"
```

### Примеры

```
Пользователь A — пришёл по ссылке:
  deep_link_param    = "instagram"
  acquisition_method = "deeplink"       ← заполняется автоматически

Пользователь B — пришёл напрямую, ответил на опрос:
  deep_link_param    = "direct"
  acquisition_source = "youtube"        ← сам ответил
  acquisition_method = "survey"         ← заполняется автоматически

Пользователь C — пришёл по ссылке И ответил на опрос:
  deep_link_param    = "ref_123456789"
  acquisition_source = "instagram"      ← сам ответил
  acquisition_method = "deeplink+survey"
```

---

## Как это работает для пользователя конструктора

Никаких новых настроек в панели свойств не нужно.
Пользователь строит флоу из существующих узлов:

```
[/start] → [Сообщение: "Как узнал о нас?"]
                ↓
    [Кнопки: Instagram | YouTube | VK | Другое]
                ↓
    [Set Variable: acquisition_source = "instagram"]
                ↓
         [Следующий узел]
```

`acquisition_source` — зарезервированное имя переменной.
Если пользователь конструктора называет переменную именно так —
она автоматически попадает в отдельную колонку `bot_users`
и отображается в аналитике как отдельная метрика.

`acquisition_method` заполняется системой автоматически:
- Есть `deep_link_param` ≠ "direct" → "deeplink"
- Есть `acquisition_source` → "survey"
- Оба → "deeplink+survey"

## Аналитика — два среза

```
По ссылкам (достоверно, deeplink):
  instagram   37%  ████████████████████
  youtube     15%  ████████
  vk          12%  ██████
  direct      36%  ████████████████████

По опросу (со слов, survey):
  instagram   45%  ████████████████████████
  youtube     22%  ████████████
  vk           8%  ████
  друг        25%  █████████████
```

Разница между срезами = люди которые пришли по ссылке одного источника
но узнали из другого. Это показывает реальную эффективность каналов.

---

## Зарезервированные имена переменных

Переменные с этими именами автоматически пишутся в отдельные колонки `bot_users`:

| Переменная | Колонка | Описание |
|-----------|---------|----------|
| `acquisition_source` | `acquisition_source` | Источник со слов пользователя |
| `utm_source` | `deep_link_param` | Алиас, для совместимости |

## Что нужно реализовать

### БД
```sql
ALTER TABLE bot_users ADD COLUMN IF NOT EXISTS acquisition_source text;
ALTER TABLE bot_users ADD COLUMN IF NOT EXISTS acquisition_method text;
```

### Шаблон `database.py.jinja2`
- Добавить `acquisition_source` и `acquisition_method` в `save_user_to_db`
- `acquisition_method` вычисляется автоматически при сохранении

### Шаблон `save-to-bot-users.py.jinja2`
- Если переменная называется `acquisition_source` → писать в колонку, не в JSONB

### Бэкенд
- Добавить поля в ответ `/api/projects/:id/users`
- Добавить агрегацию по `acquisition_source` в аналитику

### UI аналитики
- Блок "Источники трафика" показывает оба среза
- Переключатель: "По ссылкам" / "По опросу" / "Оба"

---

## Приоритет

**Low** — после `start-param-rules` и базовой аналитики.

Текущие поля `deep_link_param` и `referrer_id` покрывают 80% случаев.
`acquisition_source` нужен когда пользователи конструктора начнут
строить онбординг-опросы и захотят видеть их в аналитике.
